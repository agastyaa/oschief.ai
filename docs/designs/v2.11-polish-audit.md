## Rubric Definitions

(per `docs/designs/v2.11-solid.md` lines 375–379)

- **P0 = blocks a core task** — can't complete, silent failure, data loss risk. Example: due-date invisible → user thinks feature doesn't exist. **Must fix in v2.11.0.**
- **P1 = craft debt visible every use** — every-day friction, brand-damaging. Example: generic "No items" empty states, spinner on sub-100ms toggles. **Must fix in v2.11.0.**
- **P2 = edge case or cosmetic** — rare scenarios, minor visual. Example: 47-char name truncation edge, off-by-1px alignment in 0.1% of cases. File as **v2.12 TODOs.**

#### P2 (Destructive Actions — Hover-Only is Correct)

- **[P2] src/pages/NewNotePage.tsx:1646** — Copy/Delete transcript segment buttons, opacity-0 group-hover:opacity-100 → Contextual affordances for power users; acceptable for transcript editing
- **[P2] src/pages/Index.tsx:263** — Bulk selection checkbox, "opacity-0 group-hover:opacity-100" → Correct: toggled by selectionActive state, hover reveal is expected for bulk mode
- **[P2] src/pages/PeoplePage.tsx:493** — Checkbox/button hidden on hover → Bulk selection UI; acceptable pattern
- **[P2] src/pages/CoachingPage.tsx:368** — Border styling ("border-transparent" → "border-border") on group-hover → Subtle styling reveal; cosmetic
- **[P2] src/components/KBSuggestionsPanel.tsx:80** — Dismiss button (`X` icon) on suggestion cards, opacity-0 group-hover:opacity-100 → Decorative affordance; acceptable
- **[P2] src/components/NoteCardMenu.tsx:53** — Menu button (`MoreHorizontal` icon), opacity-0 group-hover:opacity-100 → Icon affordance is visually clear; acceptable pattern

**Verdict:** 2 P1 hover issues (primary metadata leakage). 9 P2 issues are correctly classified as destructive/decorative and safe.

### 4. Optimistic-UI Opportunities

**Scan:** Checkboxes, toggles, switches on fast operations (<100ms) that show spinners.

#### P1 (Should Be Optimistic — Currently Blocking)
- **[P1] src/pages/CommitmentsPage.tsx:84–93 (handleToggleStatus)** — Mark commitment complete/open
  - Current behavior: `await api.memory.commitments.updateStatus(commitment.id, newStatus)` → `loadCommitments()` 
  - UI waits for API call before re-rendering checkbox state
  - Problem: Local operation that feels slow; checkbox flickers on toggle
  - Fix: Update local state immediately, revalidate from server in background; re-throw error if API fails

#### Acceptable (Already Optimistic or Long Operations)
- **[OK] src/pages/DecisionsPage.tsx** — Delete button triggers `api?.memory?.decisions?.delete?.()` then re-fetches → Async, acceptable to block
- **[OK] src/pages/CoachingPage.tsx** — "Refresh" / "Reanalyze" buttons → Multi-second LLM operations; blocking is fine
- **[OK] src/pages/Index.tsx:260** — Bulk selection checkbox → Local state only, no API call; no spinner

**Verdict:** 1 P1 optimistic-UI opportunity (commitment toggle). All others are either already optimistic or legitimately async (>500ms).
