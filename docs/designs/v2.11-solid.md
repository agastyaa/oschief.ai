# CEO Plan: v2.11 "Solid" — Reliability + Polish Release

Generated 2026-04-16 via `/plan-ceo-review` (HOLD SCOPE mode, 1 round of outside-voice fixes applied)
Branch: main (future: v2.11) | Follows: v2.10 "Stabilize"
Repo: agastyaa/oschief.ai

## Premise

v2.10 makes the codebase safe to change. v2.11 makes the product feel solid.

Today OSChief is fast when everything works. When things go wrong — network drops mid-recording, Ollama crashes, user denies mic permission, force-quit during a 2-hour meeting — the experience ranges from "silent fail" to "lost data." That gap kills trust faster than any missing feature.

v2.11 closes that gap, then adds the polish layer that separates a working product from a product that feels premium. Benchmark: Granola's reliability, Linear's attention to detail, Superhuman's keyboard-first speed.

## Non-goals (explicit)

- ❌ No new features beyond polish (templates library, meeting prep briefs, weekly digest — all deferred to post-v3.0)
- ❌ No team sharing (user explicitly deferred)
- ❌ No cloud sync, no auth server (privacy-first solo stays)
- ❌ No pendant / always-on capture (wrong philosophical fit)
- ❌ No call bot / CRM push (user explicitly ruled out)

## Vision

A user hits force-quit by accident mid-meeting. Reopens OSChief. The meeting is there, partial transcript intact, ready to continue. They toggle airplane mode by accident — a small indicator surfaces, cloud LLM calls queue silently, nothing breaks. They scan the commitments list — every due date is visible at a glance, not hidden behind hover. They press `?` and see every keyboard shortcut. They never think about reliability because it just works.

## Scope (three streams, ~2 weeks)

### Stream 1: Reliability (crash/flake recovery) — ~4 days

**Each reliability item ships with a regression test.** v2.10's lesson: manual-gate-only scopes silently regress. Wave 1 adds test files alongside the code:

- `electron/main/storage/wal-mode.test.ts` — SQLite WAL mode enabled + crash-safety verification (write → kill → reopen → last committed state present)
- `electron/main/workers/worker-supervisor.test.ts` — spawn stub worker → kill → auto-restart → state preserved
- `electron/main/cloud/offline-queue.test.ts` — mock fetch failures → queue accumulates → mock reconnect → flush order correct → eviction at cap
- `electron/main/cloud/retry.test.ts` — 429/500 → exponential backoff → 401 short-circuit → max-retry respected
- `electron/main/permissions.test.ts` — permission kind taxonomy, recovery link resolution per-kind

**Named error taxonomy via `defineError()` factory** (DRY pattern — one helper, 10 one-liners; types flow via discriminated union on `error.name`):

```typescript
// electron/main/errors.ts
export function defineError(name: string) {
  return class extends Error {
    constructor(message: string) {
      super(message);
      this.name = name;
      Object.setPrototypeOf(this, new.target.prototype);
    }
  };
}

export const WALReplayError        = defineError('WALReplayError');
export const WorkerSpawnError      = defineError('WorkerSpawnError');
export const WorkerCrashed         = defineError('WorkerCrashed');
export const WorkerRestartStorm    = defineError('WorkerRestartStorm');
export const OfflineQueueFull      = defineError('OfflineQueueFull');
export const OfflineQueueExhausted = defineError('OfflineQueueExhausted'); // moved to DLQ
export const MaxRetriesExceeded    = defineError('MaxRetriesExceeded');
export const AuthShortCircuit      = defineError('AuthShortCircuit');
export const PermissionDenied      = defineError('PermissionDenied');       // attach kind via message or meta
export const PermissionRevoked     = defineError('PermissionRevoked');
export const ICloudSyncedDBPath    = defineError('ICloudSyncedDBPath');     // triggered by R1a gate
```

Each test asserts the specific class, not generic `Error`.

**P6 detail — keyboard shortcuts (Superhuman-level):**

- **Every action has a shortcut.** Record, stop, pause, new note, new meeting, mark commitment complete, snooze, assign, edit title, navigate lists, open coaching, switch tabs, trigger askbar, toggle sidebar, search, save, cancel, all of it.

- **Shortcuts are discoverable:** every button tooltip shows its shortcut. Menu items show shortcuts inline. First-run tour highlights `?` as "your keyboard map."
- **Conflict audit pass before ship:** grep for Electron menubar conflicts, OS reservations (Cmd+Space, Cmd+Tab, etc.), and vendor defaults. Document resolutions in `v2.11-keyboard-map.md`.
- **Consistency rules:** Cmd+Enter always confirms, Esc always cancels, j/k always navigate vertical lists, Cmd+K always opens palette, / always opens search in list context.
- **Measurable benchmark (10-task keyboard test suite):** documented in `v2.11-keyboard-map.md` + a manual QA checklist. Each task must be completable keyboard-only, no mouse, no trackpad:
  1. Start a new recording from empty state
  2. Stop the active recording
  3. Open the most recent meeting note
  4. Navigate to coaching tab on that note
  5. Mark the top action item complete
  6. Snooze a commitment by 1 day
  7. Open the askbar, ask a question, submit
  8. Switch to Settings → AI Models tab
  9. Toggle a setting (e.g., auto-record) and save
  10. Return to Home and search for a note by title
- Ship gate: all 10 tasks pass in a manual run on macOS. Automated Playwright test deferred to v2.12.

### Stream 5: Privacy-preserving product analytics — ~3 days

**Requested by user (2026-04-17):** need to know download count, MAU, version adoption, crash rate — without violating the "your data never leaves your Mac" promise that's OSChief's differentiator. User already has a PostHog account, so infra is zero-cost.

**Core principle:** telemetry covers product metrics (version, feature counts, crashes) only. Never meeting content, names, project data, or anything the user typed. The privacy promise applies to **meeting data**, not to anonymous product usage — but we design every touchpoint as if the user is watching.

**Design rules (non-negotiable):**
- **Opt-out = prominent.** Don't hide it in a settings submenu three clicks deep. First-run screen has two equal-weight buttons: "Share anonymous usage" and "No thanks".
- **Toggle honored immediately.** Flipping Settings → Privacy → Analytics off kills the PostHog client in the same render tick. No queued events flushed after opt-out.
- **Publish the schema.** Every event's exact JSON structure lives in `docs/PRIVACY.md`. If an event isn't documented, it isn't sent. This is the trust anchor.
- **No identity.** Install UUID is anonymous, device-scoped, uninstall-scoped. Never linked to email/name/account. `person_profiles: 'never'` in PostHog config.
- **No content.** Transcript lines, summaries, meeting titles, person names, project names, decisions, commitment text, LLM prompts, LLM outputs — never captured. Only metadata about usage (counts, durations, feature names).
- **No silent schema changes.** Schema updates ship with app versions the user explicitly installs. Never fetch analytics config from a server.

**What the user will see:**
- First-run: "Help OSChief get better? Anonymous usage stats (which features you use, crash reports) — no meeting content, no names. Change anytime in Settings → Privacy." [Share anonymous usage] [No thanks]
- Settings → Privacy → Analytics: toggle + "See exactly what we collect" link to `docs/PRIVACY.md`.

**What Sagar will see (PostHog dashboards):**

- Version adoption % (v2.10 vs v2.11 rollout curve)
- Feature usage (meetings_recorded/user/week, coaching runs, askbar queries, export actions)
- Crash rate by OS version
- Retention curves (1-day, 7-day, 30-day)
- STT engine popularity (which models people actually use)

**What Sagar will NOT see:**
- Who specifically downloaded (that's OK — GitHub download count gives aggregate numbers)
- Any meeting content, transcript, or summary text
- Names, emails, or identifiers

### Stream 6: Zero-friction runtime — remove pip install friction — ~3-4 days

**Requested by user (2026-04-17):** "Asking desktop app users to run pip commands is a significant adoption barrier." True — and it's not just speexdsp. Every Python-backed engine (speexdsp AEC, onnx-asr for Parakeet, mlx-qwen3-asr, mlx-whisper, mlx-whisper-8bit, mlx-audio for Whisper large-v3) has the same friction. Fixing them as a class in one wave.

**The friction today:** User opens Settings → AI Models → clicks "Install Qwen3-ASR" → Electron runs `python3 -m pip install --user mlx-qwen3-asr` via `execSync`. Fails on: (a) no Python, (b) Python 3.11 vs 3.12 incompatibility, (c) PEP 668 externally-managed-environment blocks, (d) corporate proxy, (e) user has homebrew Python but macOS system Python first in PATH. Each failure mode is a support ticket. Median install flow: 30-60 seconds when it works, 10-30 minutes of debugging when it doesn't.

**Ranked options:**

| # | Approach | AEC | STT engines | Effort | Tradeoff |
|---|---|---|---|---|---|
| 1 | **Bundled Python runtime** — ship `python-build-standalone` arm64 binary in `extraResources`, pre-install every Python dep at build time, point `spawn()` at bundled python. | ✓ | ✓ | L (~4 days) | +300-500MB to app. Zero user setup. One-time CI cost. |
| 2 | **Native AEC3 via `webrtc-apm` npm** — replace speexdsp worker with WebRTC's AEC3 (same algorithm that powers Chromium's `echoCancellation: true`). Node addon, no Python. | ✓ native | ✗ (still pip) | S (~1 day) | Fixes AEC only. STT deps still need pip. +2MB. |
| 3 | **Rust/C ports for each engine** — whisper.cpp for Whisper (no Python), Parakeet CoreML (already partial, finish native path), speexdsp.rs or WebRTC AEC for AEC. | ✓ | ✓ partial | XL (~10 days) | Largest perf + cleanest runtime. Way too much for v2.11 scope. |
| 4 | **Silent auto-install with robust fallbacks** — detect missing deps, silently `pip install --user --break-system-packages`, surface a toast if fallback triggers. Keep Python requirement. | ✓ | ✓ | S (~1 day) | Lowest effort. Doesn't remove Python dependency, just hides the friction more. User still needs Python 3.10+. |

**Recommended for v2.11: #2 for AEC + #4 for STT = ~2-3 days total.**

Why that split:
- **AEC is always-on** during recording with system audio. Speexdsp is a single small dep where the WebRTC AEC3 port is mature (`webrtc-apm` npm package is actively maintained, 50k weekly downloads, exposes AEC3 + NS as a Node addon). Clean native swap.
- **STT engines are opt-in and heavier.** Each engine is a different model family (ONNX, MLX, Whisper). Bundling all of them inflates app size by 300-500MB before model weights. #1 is the "right" long-term answer but overweight for v2.11. Do #4 now (silent install + better error UX), plan #1 for v3.0 when we have tighter engine defaults.

**Polish audit triage rubric (Day 4 deliverable):**
- **P0 = blocks a core task** (can't complete, silent failure, data loss risk). Example: due-date invisible → user thinks feature doesn't exist. Must fix in v2.11.0.
- **P1 = craft debt visible every use** (every-day friction, brand-damaging). Example: generic "No items" empty states, spinner on sub-100ms toggles. Must fix in v2.11.0.
- **P2 = edge case or cosmetic** (rare scenarios, minor visual). Example: 47-char name truncation edge, off-by-1px alignment in 0.1% of cases. File as v2.12 TODOs.
- **Escalation clause:** if audit finds >20 P0+P1 items, Wave 2 extends by 1-2 days rather than dropping items to P2. P0+P1 are ship-blockers for v2.11.0.

**Slip protocol (if 14 slips to 16+):**
1. Drop T8 (preload audit) — re-file as v3.0 prep work
2. Drop T7 (2 UI smoke tests) — add to v2.12 rollover
3. Compress Wave 3 by cutting T4 (stt-engine decomp) — last resort, meaningful debt
4. Never drop reliability (Stream 1) or keyboard signature feature (Stream 2 P6)
5. Never drop Zod (T1-T3) — v2.10 payoff

**Auto-update safety gate:** electron-updater check suppressed when `recording.isActive === true`. Updates apply on app quit or next cold launch when no recording is in flight. Prevents the "update-killed-my-meeting" failure mode.
